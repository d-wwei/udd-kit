
# 产品的未来：从"以用户为中心"到"用户主导开发（UDD）"

> 本文是 `UDD Kit` 的设计理念说明原文，用来解释为什么这个项目的核心会从“用户反馈”转向“用户主导问题识别、Agent 自愈与上游回流”。

## 开篇：一场真实的个体体验

自从开始深度使用 AI Agent，每天都能创造或改造各种小工具，来解决自己的问题。

一方面，日常用的工具和工作流每天都在变得更好，我深深地沉迷于这种“造物能力”。

另一方面，我发现我**再也不用依赖平台方了**，自己就可以飞速的迭代和优化。

以前使用各种产品，有不好用的地方都只能找平台方反馈，大部分是得不到回复的。
作为一名从业近十五年产品经理，我深知这背后的原因，一部分是因为研发资源有限等排期，还有一大部分需求被评估之后抛弃了。

而现在，我**再也不用找他们了**。自己使用的工具和工作流每天都可以数次迭代，而且很多产品经过改造变得更加适合我自己，再也不需要去等待平台的建设，更不用接受平台的"取舍"。

这种发生在个体身上的断裂式体验，让我作为一个互联网从业者感到深深的震撼：如果每一个普通用户都拥有了这种"造物能力"，那么基于大众需求妥协的"公司主导研发模式"必将彻底瓦解。

软件研发的终极模式，将不可逆地滑向 **“用户完全驱动（User-Driven Development）”**。或者更准确地说，是一场将造物权交接给用户的 **“用户主导开发（User-Directed Development，UDD）”**

这不是一句为了追热点而制造的口号，这意味着从需求过滤、开发权限到商业模式，整个软件生态底层逻辑的全面重构。

---

## 一、出发点：为什么会变

我们再梳理一下背景：
过去的软件开发模式之所以是"公司主导开发、用户提反馈、公司决定做不做"，本质原因并不是这种模式天然最好，而是因为过去的 **研发能力是稀缺资源**。

以前的现实是：

* 研发能力主要集中在大公司和专业开发者手里
* 软件开发成本高
* 软件迭代依赖专业团队
* 公司依靠研发能力形成产品壁垒并获得商业收益

因此形成了今天主流的软件开发流程：

1. 公司主导产品研发
2. 用户在使用中遇到问题或产生想法
3. 产品经理筛选和排序
4. 研发团队开发
5. 通过正式版本交付给用户

这个流程的核心特征是：**研发权在公司手里，用户只有表达权，没有实现权。**

但这个前提正在被打破。

随着强力大模型尤其是 Coding Agent 的出现，会彻底消除自然语言与代码实体之间的准入门槛，赋予了每一个不懂代码的普通人和业务部门"造物权"。

用户将不再愿意被动等待，而是自己直接改造实现。

## 二、辨析：为什么 Vibe Coding、SDD、Harness 都不是真正的新范式？

在深入展开 UDD 的具体含义之前，有必要先厘清一个关键区分。

现在行业内最火热的概念是 **SDD（Spec-Driven Development，规约驱动开发）**、**Vibe Coding（直觉驱动编程）**，以及诸如 **Harness 等各种令人眼花缭乱的 AI Engineering 范式**。很多人认为，既然大模型写代码的能力越来越强，未来人类只需要写好 Spec，或者凭着直觉和 AI 聊聊天（Vibe Coding），AI 就能自动生成所有的代码和复杂的工程脚手架。

但这依然是一种属于"供给侧"的技术本位傲慢。没有触及灵魂。

无论是 SDD、Vibe Coding 还是各类最新的工程范式，它们本质上都只是在优化 **"如何生产代码"** 的效率，却集体回避了一个最根本的权力交接问题：**"到底是谁在指挥这一切？"**

如果写 Spec 的、做 Vibe Coding 的人，依然是坐在核心研发室里的产品经理和企业正规军，那这种模式只不过是把传统的"人工代码砖窑"升级成了"体验更丝滑的自动化代码工厂"。

有人会说：成本降低了，我们终于可以用 AI 覆盖更多长尾需求了。没错，但问题不在于你能不能覆盖，而在于你的覆盖速度永远追不上需求产生的速度。用户的痛点是即时的、情境化的、千人千面的——当一个运营在周二下午三点需要"把本周新增客户按省份拆成十张表并自动发给对应区域负责人"时，他不会提一个需求单然后等你下个迭代。他的 Agent 三分钟就做完了。

这才是根本性的区别：不是"谁写代码"的问题，而是"谁发起开发、谁定义需求、谁决定优先级"的权力归属问题。当用户自己就能即时解决问题时，他为什么要等你？

真正的重构，不在于代码由谁生成，也不在于开发体验有多么无缝（Vibe），而在于开发权从"供给端（公司）"彻底转移到了"需求端（用户）"。

因此，比现有的所有效率范式都更准确、更具有革命性意义的，应该是 **UDD（User-Directed Development，用户主导开发）**。
这不是只属于工程师的代码革命，而是关于软件权力下放的终局。

厘清了这一点，我们接下来具体拆解 UDD 带来的几个核心变化。

## 三、被颠覆的需求漏斗：从"最大公约数"到"无限扩展的个性化"

先来一场产品经理的自我审视，过去我们做的最核心的一件事，就是判断：

* 哪个需求是真需求，哪个是伪需求
* 哪个需求用户量足够大，哪个优先级更高
* 哪个需求要排进 roadmap，哪个要被砍掉

从成千上万个需求中，提炼出一个"最大公约数"。减法做得好不好，非常能体现出产品经理的水平。

这并非出于傲慢，而是因为**算力与研发能力极度稀缺**。企业被迫将成千上万个个性化痛点无情剔除，提炼成一个能满足"最大用户和商业价值"的标准功能，研发交付后，再让所有人去适应这个充满了妥协的标准版。

换句话说，过去需求必须经过一个"通用化压缩"的过程：

* 原始需求是千千万万充满个性化的
* 而经过抽象提炼——
* 只有足够多人需要、足够有商业价值的需求，才值得被正式开发

但是未来，**很多需求根本不需要被抽象**

因为每个用户都可以针对自己的使用习惯、目标、偏好，去开发属于自己的功能。于是：

* 不需要再特别强调需求的通用性
* 不需要再强行抽象成"标准产品能力"
* 不需要为了兼容所有用户而牺牲个性化
* 不需要因为用户少就排除掉某个需求
* 不需要为了"优先级"而推迟小众但真实的需求

这意味着软件开发将从：

**围绕大众需求优化**

转向：

**允许无限长尾的个性化定制**

也就是说，未来的大部分软件都不再有标准形象，而是：

> 一个稳定核心基座上，叠加无数用户自己生成的个性化扩展。

这背后的变化不是"功能更多"，而是：

**软件从面向群体的统一产品，变成面向个人的持续定制系统。**


## 四、公司角色的变化：从"全栈开发者"变成"平台与框架提供者"

由上面的逻辑继续往下推，公司要做的事情也会变化。

未来一家公司不必再把所有能力都牢牢抓在自己手里开发。更合理的方式可能是：

* 把产品结构做得足够模块化
* 把功能拆得足够原子化
* 定义清晰的扩展边界
* 把核心能力封装好
* 允许用户或用户 Agent 在外围进行安全扩展

最重要的几件事变成了：

### 1. 设计最初的核心逻辑与框架

公司仍然要负责最底层的核心架构，包括：

* 核心产品逻辑
* 数据结构
* 能力边界
* 权限模型
* 接口约束
* 审计与回滚机制

### 2. 把系统设计得适合被扩展

也就是让系统本身具备：

* 模块化
* 插件化
* 原子化功能单元
* 标准化接口
* 可沙盒执行的扩展层

### 3. 负责调度、审核、协调

如果大量扩展由用户或 Agent 来完成，那么公司最核心的职责之一会变成：

* 审核扩展
* 协调冲突
* 维护生态秩序
* 做安全审计
* 保证系统一致性

因此，公司未来更像：

**平台运营者、协议制定者、核心系统维护者、能力市场组织者**

而不再只是"唯一开发者"。

-----

说到这里，也许你会想：这不就是让用户自己做插件吗？做好 API 让他们接入不就行了？再或者搞一个高级版的油猴脚本/开源社区？但我们的核心产品又不是开源的，用户怎么改？

接下来我们来回答这两个问题。

## 五、UDD 与传统 API 插件模式的本质区别

"做一套 API 让大家去开发"听起来是对的，但它依然停留在一个旧时代的陷阱里：**权力的本质依然在供给侧。**

传统插件生态有三个无法跨越的硬伤，这也正是 UDD 时代要爆发式颠覆的地方：

1. **边界定义的权力倒转（API 是封建分封，UDD 是丛林拓荒）**

**老插件模式：** 官方必须首先穷举、设计并开放出这套 API。官方没想到的，或者处于安全考虑不愿意开的口子，用户绝对做不了。用户依然是在官方画好的沙坑里玩沙子。

**UDD 新范式：** 你不给 API？没关系。强力 Agent 拥有视觉理解、DOM 树解析、系统接管的能力。当一个不懂代码的销售让 Agent 去"抓取昨天的订单并私信发给未付款客户"时，Agent 是不会去等官方发版 API 的，它会像个"赛博人"一样直接暴力拼装出工作流。**不是你提不提供 API 的问题，而是如果官方不主动提供安全扩展的框架，你的产品底座就会被无数野生 Agent 用"外挂"的方式撕碎。**

2. **颗粒度与成本的指数级差异（插件是项目开发，UDD 是"极窄定做"）**

**老插件模式：** 开发一个插件是正规的"软件项目"，只解决"某一群人"的共性需求（如抢票插件），因为只有受众多，才有开发者愿意投入时间去读文档、写代码、排期上架。

**UDD 新范式：** 开发成本降到了趋近于 0（一句自然语言）。于是，需求下探到了几十亿个**"一次性、个人化、即刻生成、用完即抛（Intent-to-Dispose）"**的纳米级维度。没人会为"帮我把表单里名字带'李'的数字变红"去上架一个插件，但 UDD 模式下它只需要 3 秒钟。

3. **维护者身份的根本性转变（从"死代码"到"系统自愈"）**

**老插件模式：** 官方一旦大版本更新改了底层接口（API Break），历史插件会瞬间"尸横遍野"，引发大量报错。

**UDD 新范式（Auto-Healing）：** 每一个扩展背后不再是一个人类开发者，而是一个永生的 Agent。当官方发布了新的系统规约，用户的 Agent 会在后台自我唤醒，自动读取新规约、重写逻辑并静默自我修复。系统的演进彻底抛弃了被"向前兼容"绑死的沉重包袱。

举个例子：某个 CRM 平台升级了客户字段结构，从扁平字段改为嵌套对象。在传统插件模式下，所有依赖旧字段结构的第三方插件会瞬间报错，开发者要么已经离职、要么排期遥遥无期。而在 UDD 模式下，用户的 Agent 检测到 schema 变更后，自动读取新的字段映射规则、重写查询逻辑，整个过程在用户无感知的情况下完成。


## 六、闭源软件的解法：Sandbox AI 与零知识交付

商业项目往往不会开放核心代码，因为那是核心资产和商业壁垒。那用户要怎么在不接触核心代码的前提下进行功能开发？

我们尝试提出一个答案，闭源软件也可以实现类似开源式创新，但方式不是直接开放源代码，而是开放"可控开发环境"和"受限扩展层"。

这是整套讨论里最有原创味道的机制之一：**沙盒架构（Sandbox AI Development）**与**零知识交付（Zero-Knowledge Delivery）**。

### 基本流程

1. 用户提出一个个性化需求
2. 前端 Agent 理解需求并将其结构化
3. 系统为这个需求创建一个独立的沙盒环境 / 临时仓库
4. 后端研发 Agent 在沙盒中开发功能
5. 开发完成后，将功能交付给用户试用
6. 用户给出反馈
7. 继续迭代
8. 必要时将其升级为可长期使用插件，或提交为平台生态的一部分

这样的效果是：
1. 用户会得到想要的功能，但拿不到代码
2. 不需要真正开源，就能让外部创新发生

你可能会说，即使把前后 Agent 分开，用户仍然可能通过：

* 对前端 Agent 的诱导
* 提示词注入
* 反复套话
* 需求伪装
* 上下文污染

来试图获取内部信息。

这里除了 Agent 之间的通信边界、行为权限要求、可见信息范围的硬约束外，我们提出一个更强的安全方法：**双 Agent 的零知识交付空间**。

### 零知识交付的核心原理

当后端沙盒里的研发 Agent 要向前端用户代理交付功能时，不要直接常驻通信，而是创建一个 **临时的、任务级的、执行后即销毁的安全空间**。

这套机制的关键在于三层隔离：

**第一层：信息不对称隔离。** 前端 Agent（用户代理）只知道用户需要什么功能、期望什么效果，但完全看不到后端代码库的结构、实现细节和内部接口。后端研发 Agent 可以访问核心代码，但只能通过结构化的"需求规约"了解用户意图，无法直接与用户对话。两个 Agent 之间不存在自由通信通道。

**第二层：交付物脱敏。** 后端 Agent 开发完成后，产出物经过一个自动化的"脱敏编译"流程——将沙盒中的源码编译、打包为不可逆的运行时制品（如 WASM 模块、加密容器镜像或受控 API 端点）。用户拿到的是一个可以运行的"黑盒功能"，而非可阅读的源代码。

**第三层：执行后即焚。** 每一次开发任务结束后，沙盒环境连同其中的上下文、中间产物、调试日志一并销毁。没有任何一方能事后回溯沙盒内发生过什么——唯一留存的是最终交付给用户的功能制品本身。

你可以理解为：就像一个梦，梦里发生的一切你都不记得，但是梦醒了你手边多了一件在梦中锻造好的工具。

依靠这种"表面无限扩展，底层绝对物理隔离"的模式，闭源老厂也能在极其安全的环境中，爆发出碾压开源社区的极客级长尾创新。


## 七、企业的选择：只有更主动地向用户提供定制开发能力

站在商业软件厂商的角度，很多人会警惕：既然用户连逆向工程都敢做，我们是不是应该把安全墙筑得更高？为什么要主动为用户提供定制环境？

答案非常残酷：**防守是无效且致命的。只有主动下放权柄，才能成为新时代的平台入口。**

**1. 消费级（ToC）应用的心智与入口保卫战**

对于零售、内容社区等消费级 App，主动拥抱 UDD 是防御 Agent "剥夺视觉界面（Headless 化）"的唯一手段。

用户的 Agent 永远会找到外挂、爬虫和无头浏览器（RPA）等方式去替用户执行任务。如果平台不主动提供官方支持的 Agent 定制能力（比如让用户自行组装定制化的内容过滤流或比价工具），用户的通用 Agent 就会在应用之外进行拦截和接管。

届时平台将丧失所有前端交互甚至广告曝光价值，沦为一条只提供基础数据的冰冷管线。主动拥抱 UDD，是把未来极度个性化的人机交互过程强留在自己体系内（In-App）的底线。

**2. 企业级（ToB）厂商跳出"二开"无底洞**

无论是 SaaS 厂商服务大企业（KA），还是传统软件进行私有化部署，最终都会被无穷尽的"二次开发"定制需求活活拖死。主动交出组装权，让客户的系统与 Agent 去各自完成长尾定制，是原厂研发团队跳出"驻场外包化泥潭"的唯一解。

**3. 生态维度的降维打击**

试想市面上有两款同类软件。一款是封闭黑盒，所有需求都需恳求官方排期；另一款主动提供受控沙盒与 AI 工具链，客户能在半天内让自己的 Agent 定制出一套前沿机制。采购决策的结果不言而喻。

因此，主动拥抱"兵权下放"，是软件厂商顺应新范式、延续平台统治力的最优解。

---

## 八、平台的新职责：定义规则边界与拦截"恶意诉求"

当把开发权下放给海量的个体 Agent 后，作为产品提供方，我们要面临一个新的严峻挑战：**如果用户的定制需求是恶意的、或者是破坏系统公平性的，该怎么办？**

例如：
* 某个用户利用 Agent 生成了一个抢单插件，要求"我的请求优先级永远比别人高，比别人快"。
* 一个业务员生成了一个能够高频越权抓取全公司销售线索的脚本。
* 客户的代理无底线地消耗平台的公共计算资源。

商业公司当然可以有不同的用户服务等级（比如付费专享加速通道），但这必须是在官方定义好的规则之下，而不能允许用户的 Agent 去做野生劫持。

因此，在这个新范式里，平台方（甚至产品经理）并没有消亡，只是职责从"做加减法"，转向了**成为新造物生态的规则制定者（Rule Maker）和边界守夜人**。

平台必须引入**强力的行为审计与自动化审查（Automated Auditing & Filtering）**：
* 设定清晰的安全隔离边界（沙盒机制的必要性）。
* 明确甄别出哪些定制是"属于自身效率提升"，哪些定制是"跨越红线的公平性破坏（零和博弈）"。
* 当恶意插件试图上线时，在沙盒层面能够基于意图将其熔断。

"把开发权交还给用户，并不等于交出系统的秩序权。"

接下来，我们从商业视角审视 UDD 带来的终局版图。

## 九、深不可测的用户粘性：从"数据绑架"到"外在器官"

在过去的标准化软件时代，用户之所以难以离开一个产品，主要是因为：
* 数据迁移成本高
* 团队协作绑定
* 学习成本

但在 AI 时代，这些传统的壁垒正在被削弱（数据可以被一键导出清洗，AI 可以快速教会用户使用新软件。比如 Anthropic 迁移了几十万 OpenAI 的用户数据）。

而在 UDD 的范式下，真正不可逾越的护城河变成了：**持续生长的个性化资产与深刻的习惯积累。**

未来的软件再也不是标准化的"出厂设置"。当企业或个人在使用中，他的 Agent 不断为他在沙盒里生成专属功能、定制了他的使用路径、重塑了他的界面，甚至这些定制化功能还在随着他的使用**持续变化和生长**……

那么这个产品最终就变成了 **"带着用户DNA的外在器官"**。

* 这里的每一个能力都是按照他的偏好定做的
* 这里积累了他无数次微小的行为矫正
* 沉淀了他乃至他整个团队"最顺手"的神奇组件

对于这样的产品，用户的转移成本几乎是无限大的。因为如果要整体迁移到另一个软件，他失去的不仅是冷冰冰的数据，而是**"一个完美适配自己灵魂形状、与自己共同进化出来的定制化系统"**。这样的粘性，比过去任何软件应用的绑定都要深厚得多。

有人可能会质疑：如果 Agent 能帮你在 A 平台上从零搭建一套完美定制系统，同样的 Agent 不也能帮你在 B 平台上重建吗？理论上是的，但现实中有两个关键阻力。第一，定制资产不仅仅是代码逻辑，更包含了数以千计的微小行为校准和隐性偏好积累——这些是长期使用中沉淀出来的，无法通过一次性迁移完整复制。第二，即使 Agent 能重建功能框架，在新平台上重新磨合到同样的流畅度，需要的时间和精力成本依然高昂。这就像搬家可以搬走家具，但搬不走你对每一个角落的熟悉感。

## 十、网络效应的新来源：用户越能开发，平台越强

过去网络效应的增长通常依赖于"更多用户、更多数据、更多内容"。但在 UDD 时代，如果每个用户都可以开发自己的功能，公司的网络效应将被史无前例地放大。这源于两种全新的效应：

**1. 更多用户带来更多能力创新**
过去的软件生态里，用户仅仅是消费端或反馈端。但在 UDD 之下，用户直接转变为"扩展能力的生产者"。
于是一个平台会越来越像一个巨大的**插件市场、技能市场与 Agent 协作网络。**
这意味着，用户数量的增长不再仅仅带来订阅费或 DAU 的线性增长，更会带来：
* 更多功能组件
* 更多 Agent Skill
* 更多工作流模板
* 更多跨场景能力组合

最终形成一个无可匹敌的正反馈飞轮：**用户越多 → 平台沉淀的定制能力越多 → 平台越有吸引力 → 又吸引更多新用户。**

**2. 衍生出基于信任与声誉的 A2A 网络（Trust & Reputation）**
当数以成千上万的用户凭借 Agent 极低成本地生成海量私有组件并开始交流借调时，平台便升维成了一张 **A2A（Agent-to-Agent）Network**。
组件的泛滥不可避免地会导致市场充斥同质化甚至恶意的代码。因此，未来的平台将在此之上自然生长出"面向机器与 Agent 的信誉系统"。依靠诸如沙盒行为历史画像、安全评级、资源消耗指数等数据，依据行为评判发放信用分。
这让 Agent 在跨域编排和调用底层能力时，能够依靠平台进行风险预测和路由选择。能建立并掌控这套**"Agent 间信任评价与路由大坝"**的平台，将拥有云原生时代最深不可测的护城河。

## 十一、新的营收空间：云托管与基于"云和地租"的新模式

旧范式下，我们靠传统的软件买断、内购或基础的会员费来赚取订阅收入；但在 Agent Native 且一切皆可由用户定制（UDD）的时代，成千上万的用户在不断创造新的扩展能力。这就引出了一个极其现实的核心问题：**海量定制能力全天候运行带来的庞大算力和服务器成本，谁来承担？**

这个问题不仅不是灾难，反而恰好构成了未来平台最丰厚的商业基石。如果主动将其转化为基础设施服务，就会演化出两种并行的运行模式——这也正是未来平台最核心的变现路径：

### 1. 本地免托管模式（即装即用）
这是最轻量化的一种延伸生态。
* **适用场景**：个人的简单自动化脚本、低资源需求的过滤规则等。
* **特点**：直接在用户本机的沙盒或浏览器端侧运行，无需云托管，即装即用。公司不需要承担额外的服务器运算成本，这部分流量完全属于赋能生态的长尾补充。

### 2. 平台云托管模式（云管端一体化收租）
这才是未来软件平台的营收主力。
* **适用场景**：当用户的专属组件涉及复杂的后端逻辑、需要长时在线监听事件、处理海量业务数据，或需要多终端（手机/PC）严格同步一致的体验时。
* **机制与打法**：平台主动在云端提供一套"安全、受控且计费的专属沙盒托管环境"。它包揽了部署、高可用运行和资源监控。用户只需将自己写好的极复杂 Agent 组件"一键 Push 到云端"，即可享受 7x24 小时的稳定执行。

于是，**传统的软件开发商（SaaS公司）实际上正在向半个云厂商（AWS）与能力集市杂糅的形态转变**。
未来的商业触角将遍布"造物"与"化身执行"的每一个环节，形成极为多元的新一代收租体系：
1. **云端托管与算力费**：针对在云端沙盒中运行的组件，收取涵盖计算（Compute）、存储（Storage）与带宽（Bandwidth）的资源托管费。
2. **底层大模型材料费**：系统内置大模型的 API 调用、各类 Token 消耗费用打包结算。
3. **高阶工具与专属房租**：提供给进阶开发者、重度用户的云端专属 AI Studio 席位费或更强力的物理隔离沙盒订阅费。
4. **护航保障及企业服务费**：向大企业客户提供私有化或独占级别的 SLA 保障支持方案。
5. **能力过路费与数字地租**：在 A2A（Agent-to-Agent）的网络中，平台凭借那套**"面向机器的信誉测评与路由分发系统"**，在此之上发展出天然的插件/技能交易市场抽成方案。在每一次跨用户能力借调与插件订阅中，平台都能顺理成章地抽取"能力撮合费"。

**最终，未来的云原生软件巨头，赚取的将绝不仅是那一点极易被取代的软件本体订阅费。他们将像提供水电煤等基础设施的基建狂魔一样，在一个极度繁荣的个体数字贸易生态中，稳稳收取源源不断的云端算力费与商业地租，构建出云原生时代无与伦比的商业护城河。**

## 终局点题：重塑航海图

> **Software for humans and their agents — extended by users, co-evolving with habits, governed by platform boundaries, developed with AI, and protected by sandbox.**
>
> **同时为人和 Agent 而生的软件——由用户扩展，与习惯共同进化，受平台边界治理，以 AI 驱动开发，在沙盒中守护安全。**

在过去的二十年里，产品经理想尽方法"教育"我们的用户如何适应我们画好的交互地图。
从现在起，软件开发的历史被彻底改写。

**软件的未来不再由公司的路线图定义，而由每一个用户——和他背后的 Agent——亲手塑造。** 这就是用户主导开发的终局：人决定方向，Agent 执行建造，平台退守为边界与土壤。
